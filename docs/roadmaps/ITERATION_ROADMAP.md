# Crazy Factory 程序迭代路线图

> 目标：把“功能想法”转化为“可排期、可验证、可交付”的程序迭代计划，确保开发节奏稳定推进。

## 1. 迭代目标（Program Iteration Goals）

- **稳定性优先**：先保证核心循环与存档安全，再叠加新系统。
- **增量交付**：每个迭代必须产出玩家可感知变化。
- **可验证落地**：每个迭代绑定脚本验证与验收标准。
- **可回滚发布**：变更具备回退路径，避免坏档和不可逆改动。

---

## 2. 迭代节奏与周期

- **建议周期**：2 周 / 1 个迭代（10 个工作日）。
- **阶段分配**：
  - Day 1-2：需求拆解 + 技术设计 + 验收定义
  - Day 3-6：开发实现 + 单轮自测
  - Day 7-8：平衡回归 + 边界修复
  - Day 9-10：稳定性回归 + 发布准备 + 复盘

---

## 3. 六次迭代规划（建议）

## Iteration 1：经济核心与存档安全 ✅
**目标**：夯实经济计算与存档迁移基础。

**交付项**
- 经济核心函数边界检查（价格、收益、离线、Prestige）。
- 存档版本迁移与回退策略校验。
- 参数配置结构整理（便于后续调参）。

**交付明细（落地清单）**
- 经济边界样例：至少覆盖“零值、极值、长时离线、Prestige 临界点”。
- 存档迁移样例：至少覆盖“当前版本、上一个版本、异常字段缺失版本”。
- 参数结构整理：核心经济参数集中定义，并附参数变更记录入口。

**验收标准（DoD）**
- 关键公式脚本回归通过。
- 老存档可识别并安全迁移。
- 本迭代最小验证命令执行成功：
  - `node scripts/economy_checks.js`
  - `node scripts/baseline_15m.js`
- [报告] `ITERATION_1_REPORT.md`

## Iteration 2：Prestige 2.0 与成长分支
**目标**：将 Prestige 从“重置”升级为“策略决策”。

**交付项**
- 至少 2 条永久成长分支。
- Prestige 后 5-10 分钟内可感知差异。
- 对应经济断言与回归样例补齐。

**验收标准（DoD）**
- 不同分支存在可复现收益差异。
- 不出现单一路线长期碾压。

## Iteration 3：订单高阶池与产业链联动
**目标**：提升中期策略深度与循环变化。

**交付项**
- 新增高阶订单模板（含金融/产业链标签）。
- 订单奖励与成本收益比调优。
- 完成演出反馈与奖励日志记录。

**验收标准（DoD）**
- 10~20 分钟出现 3 次以上有效决策点。
- 刷新订单成本可恢复，不形成惩罚锁死。

## Iteration 4：金融策略档位与风控系统
**目标**：建立“稳健/平衡/激进”可切换的金融玩法。

**交付项**
- 策略档位按钮与参数包。
- 风险回撤约束与收益上限治理。
- 金融建议动作提示（投入/提取）。

**验收标准（DoD）**
- 三种策略均可推进至首轮 Prestige。
- 激进策略存在高波动与回撤窗口。

## Iteration 5：模块化拆分与工程治理
**目标**：降低耦合，提升可维护性与开发效率。

**交付项**
- 从 `index.html` 抽离核心模块（`economy.js` / `save.js` 等）。
- 建立模块边界与依赖约定。
- 补齐关键行为一致性检查。

**验收标准（DoD）**
- 功能改动局部化，不影响主循环。
- 模块改造后基线体验不回退。

## Iteration 6：发布准备与版本演练
**目标**：形成可持续发布流程。

**交付项**
- 版本号规范与发布清单。
- 回滚机制与异常处理 SOP。
- 发布演练（至少 1 次完整流程）。

**验收标准（DoD）**
- 发布流程可重复执行。
- 发生问题时可在可控时间内回滚。
