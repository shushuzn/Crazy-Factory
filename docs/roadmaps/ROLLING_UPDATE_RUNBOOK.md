# Crazy Factory 滚动更新执行手册（Runbook）

> 目标：把“滚动更新监控路线图”转成可直接执行的发布操作手册。

## 1. 发布前（Pre-Release）

- [ ] 确认版本号与变更说明。
- [ ] 确认本次变更是否涉及：经济核心、存档结构、Prestige、订单与金融参数。
- [ ] 执行并记录必跑脚本：
  - `node scripts/economy_checks.js`
  - `node scripts/baseline_15m.js`
  - `node scripts/balance_runs_20m.js`
  - `node scripts/stability_30m.js`
  - `node scripts/finance_checks.js`
- [ ] 输出发布前摘要：通过/失败项、风险点、回滚触发条件。

## 2. 扩量节奏（Rollout Stages）

## Stage 1：10%
- 观察窗口：10-15 分钟。
- 关注指标：运行错误、存档迁移失败、NaN/Infinity。
- 放量条件：无 P0/P1 告警，关键体验指标未明显恶化。

## Stage 2：25%
- 观察窗口：15-20 分钟。
- 关注指标：错误率趋势、回滚率、异常反馈量。
- 放量条件：错误率稳定，未出现版本特异性异常聚类。

## Stage 3：50%
- 观察窗口：20-30 分钟。
- 关注指标：经济指标偏移（gps/lifetimeGears）、订单完成率。
- 放量条件：业务指标在阈值区间内，无持续性异常。

## Stage 4：100%
- 观察窗口：30 分钟。
- 关注指标：全量稳定性、核心漏斗指标回归。
- 完成条件：进入“发布后复盘”并沉淀异常处理记录。

---

## 3. 告警与处置（Incident Handling）

## P0（立即回滚）
触发条件（任一）
- `economy_checks` 失败。
- 稳定性出现 NaN/Infinity。
- 存档迁移出现不可恢复故障。

处置动作
1. 立即暂停扩量。
2. 执行版本回滚。
3. 发布事故通告（影响范围 + 时间线 + 下一步）。

## P1（暂停扩量并修复）
触发条件（任一）
- 存档迁移失败率 > 0.5%。
- JS 错误率环比翻倍并持续 10 分钟。

处置动作
1. 保持当前批次不扩量。
2. 组织工程/数值联合排查。
3. 决定热修或回滚。

## P2（记录并纳入下个版本）
触发条件
- 轻微体验退化但未越线。

处置动作
1. 记录问题与复现路径。
2. 纳入下个迭代修复计划。

---

## 4. 回滚流程（Rollback SOP）

1. 确认触发条件与影响范围。
2. 切换流量到上一稳定版本。
3. 执行回滚后验证脚本：
   - `node scripts/economy_checks.js`
   - `node scripts/stability_30m.js`
4. 确认关键指标恢复（错误率、存档成功率、NaN/Infinity=0）。
5. 产出事故复盘（24 小时内）。
