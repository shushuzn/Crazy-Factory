# Crazy Factory 滚动更新监控路线图

> 目标：建立“发布中可观测、异常可告警、问题可回滚”的滚动更新监控体系，降低版本发布风险。

## 1. 监控目标
- **发布可见性**：每次滚动更新可实时查看进度与健康状态。
- **质量可量化**：核心经济、存档、性能指标有统一阈值。
- **异常可处置**：触发告警后有明确分级与处置流程。
- **回滚可执行**：达到止损条件时可快速回滚到稳定版本。

---

## 2. 监控分层（What to Monitor）

## A. 发布层（Rollout Health）
- 发布批次进度（10% → 25% → 50% → 100%）。
- 各批次错误率、失败率、回滚率。
- 各批次用户反馈异常数（崩溃/卡死/坏档）。

## B. 业务层（Game KPIs）
- 关键经济指标：`gps`、`lifetimeGears`、订单完成率。
- 关键体验指标：10 分钟有效操作率、首次 Prestige 达成率。
- 关键稳定指标：NaN/Infinity 出现率、存档加载失败率。

## C. 技术层（Runtime & Save）
- JS 运行错误数量与堆栈聚类。
- 存档迁移成功率、回退触发率。
- 长时运行性能（30 分钟内帧率/内存趋势）。

---

## 3. 分阶段建设计划（4 周）

## Week 1：监控口径与阈值定义
**交付项**
- 指标字典（名称、口径、采样频率、阈值）。
- 告警分级（P0/P1/P2）与通知链路。
- 发布看板草图（发布层 + 业务层 + 技术层）。

**验收标准（DoD）**
- 核心指标不少于 12 项且均有阈值。
- 告警分级与负责人明确。

## Week 2：最小可用监控上线（MVP）
**交付项**
- 发布进度看板（批次、错误率、回滚率）。
- 关键脚本结果自动记录：
  - `node scripts/economy_checks.js`
  - `node scripts/baseline_15m.js`
- 手工周报模板（发布结果 + 异常摘要 + 处置记录）。

**验收标准（DoD）**
- 每次发布后 30 分钟内可输出监控摘要。
- 脚本结果可追溯到具体版本号。

## Week 3：异常联动与止损机制
**交付项**
- 告警触发规则与升级策略（连续失败/阈值越线）。
- 回滚决策表（何时降批、何时暂停、何时全量回滚）。
- 发布故障演练（至少 1 次）。

**验收标准（DoD）**
- 告警到处置路径可在 10 分钟内启动。
- 回滚流程可在演练中完整跑通。

## Week 4：滚动更新标准化
**交付项**
- 标准发布清单（发布前/发布中/发布后）。
- 复盘模板（根因、影响范围、修复计划、防复发措施）。
- 月度趋势分析（错误率、回滚率、稳定性得分）。

**验收标准（DoD）**
- 连续 2 次发布按清单执行并完成复盘。
- 关键故障指标趋势可视化并可解释。

---

## 4. 告警阈值建议（初版）
- `economy_checks` 任一失败：**P0（立即阻断发布）**。
- 30 分钟稳定性出现 NaN/Infinity：**P0（立即回滚）**。
- 存档迁移失败率 > 0.5%：**P1（暂停扩量，进入修复）**。
- 关键 JS 运行错误率环比翻倍：**P1（降批并观察）**。
- 体验指标轻微回落但未越线：**P2（记录并下版本修复）**。

---

## 5. 每次滚动发布必跑清单
```bash
node scripts/economy_checks.js
node scripts/baseline_15m.js
node scripts/balance_runs_20m.js
node scripts/stability_30m.js
node scripts/finance_checks.js
```

---

## 6. 角色分工（建议）
- **发布负责人**：控制扩量节奏与回滚决策。
- **工程负责人**：技术异常定位与修复推进。
- **数值/策划负责人**：经济与体验指标异常分析。
- **项目负责人**：跨角色同步、对外风险沟通与复盘收敛。
