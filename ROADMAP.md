# 自动化工厂（Crazy Factory）项目化路线图（Vibe Coding 版）

> 目标：把当前单文件原型推进到“可持续迭代 + 可验证平衡 + 有明显游戏感”的小型可发布版本。

## 1) 项目目标与成功标准

### 产品目标（What）
- 打造 1 个可持续游玩 20~40 分钟的放置游戏竖切版本。
- 在前 3 分钟建立“点击爽感 + 自动化目标感”，在 10 分钟后给到中期策略选择（订单、技能、Prestige）。
- 保持低维护成本：核心经济逻辑可测试、UI 反馈可配置、功能迭代可控。

### 成功标准（How to know）
- **留存体验代理指标**
  - 新玩家 3 分钟内完成首个阶段任务（目标达成率 ≥ 80%）。
  - 10 分钟内至少触发 1 次“强反馈时刻”（成就/订单/Prestige）达成率 ≥ 70%。
- **经济稳定性指标**
  - 任意 15 分钟流程内无“资源断崖”（长于 90 秒无可操作项）。
  - Prestige 回圈后，回到重置前产能耗时应显著缩短（建议 ≤ 40%）。
- **工程质量指标**
  - 核心公式（价格、离线、奖励）具备最小自动化校验。
  - 长时间运行（30 分钟）无明显性能抖动或内存膨胀。

## 2) 里程碑分期（按可交付物）

## M1：核心经济与结构稳态（1~2 次迭代）
**目标**：把“能跑”提升到“可调、可测、可维护”。

### 交付物
- [ ] 经济层函数收敛：价格、收益、离线结算从渲染逻辑中抽离为纯函数。
- [ ] 引入最小测试脚本：覆盖 `price/gps/offline/prestige` 的关键边界。
- [ ] 存档结构版本号（`saveVersion`）与兼容分支。
- [ ] 关键参数集中配置（倍率、阈值、反馈时长）并统一命名。

### 验收标准
- [ ] 修改任意经济参数，不需要改 UI 逻辑。
- [ ] 公式回归测试可一键跑通。
- [ ] 老存档导入失败时有可读提示，不会直接坏档。

---

## M2：玩法层扩展（中期目标 + 策略选择）（2~3 次迭代）
**目标**：形成“短期操作 - 中期目标 - 长线回圈”的稳定节奏。

### 交付物
- [x] 生产订单系统（解锁、进度、领取、刷新）
- [ ] 订单池扩展：按阶段解锁不同订单类型（点击/产能/建筑组合/限时目标）
- [ ] 建筑专属升级分支（每类建筑至少 2 条差异化路线）
- [ ] Prestige 2.0：新增 1~2 个永久被动（非纯数值乘区）

### 验收标准
- [ ] 玩家可在“直接扩产”与“订单刷新”之间做出有意义选择。
- [ ] 10~20 分钟流程至少出现 3 次不同决策点。
- [ ] 不同策略在收益上接近（避免单一最优解碾压）。

---

## M3：Game Feel 与表现层升级（1~2 次迭代）
**目标**：把反馈从“有”提升到“爽”，但不牺牲性能。

### 交付物
- [x] 轻量反馈基础：浮字、按钮弹跳、面板脉冲
- [ ] 连击反馈：连续点击触发递增视觉/音效层级
- [ ] 关键时刻反馈：任务完成、订单完成、Prestige 触发短时强调动画
- [ ] 音频总线雏形：SFX 音量与开关、事件触发点统一入口

### 验收标准
- [ ] 所有高价值动作（购买、升级、领取、重置）都有明确反馈。
- [ ] 反馈参数可在常量区 1 处调整（时长/强度/频率）。
- [ ] 无“反馈刷屏”问题（浮字数量、动画并发有上限）。

> 性能提醒：当前浮字为 DOM 动态创建，后续若频率继续增加，建议改对象池或 Canvas 粒子层，减少频繁分配与回收。

---

## M4：工程化与发布准备（2 次迭代）
**目标**：完成“可演示版本”到“可发布版本”的最后一公里。

### 交付物
- [ ] 文件拆分：`economy.js / progression.js / ui.js / save.js`。
- [ ] 调试面板：倍率覆写、资源注入、存档快照导出。
- [ ] 平衡配置外置（JSON）+ 一键导入。
- [ ] 发布资产：更新日志模板、版本号策略、基础 FAQ。

### 验收标准
- [ ] 主逻辑不再集中在单文件。
- [ ] 数值迭代无需改核心逻辑。
- [ ] 每次发布有可追踪变更记录。

## 5) 风险与应对
- **风险：数值膨胀导致玩法失真**  
  应对：分层乘区、设置软上限、每次迭代只改 1~2 个关键参数。
- **风险：反馈过多影响性能或造成视觉疲劳**  
  应对：反馈分级（普通/关键）、并发上限、低配降级开关。
- **风险：单文件继续膨胀导致改动牵一发动全身**  
  应对：先抽纯函数，再拆模块；每拆一步都保留可运行版本。

---

## 6) 更新规则
- 每次迭代只更新三块：`当前状态`、`两周执行清单`、`风险与应对`。
- 每个勾选项必须对应一个可验证结果（截图、日志或脚本输出）。
- 保持短句、可勾选、可追踪；避免“愿景化空话”。
